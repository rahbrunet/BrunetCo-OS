# Agent Roster — A0 through A18

Nineteen agents, all operating under one non-negotiable v1 rule: **agents draft and propose; humans approve anything that gets sent, filed, invoiced, instructed, or posted.** Where an agent's design includes a path toward more autonomy later, that path is built but kept dormant (see A18 below for the clearest example).

---

## A0 — Orchestrator

The chat interface and control layer that everything else sits under: tool registry, approval queue, audit logging, the confidentiality egress gate (encrypted LLM calls, ZDR terms where available), and Bitwarden runtime credential fetching. A0 also owns two capabilities that aren't separate agents in their own right: the **Prospect Engagement Orchestration** journey (§ below) and **unscripted-project planning** (the natural-language-to-project-plan capability described in the M9 file). Think of A0 less as "one more agent" and more as the substrate the other eighteen run on.

## A1 — Status Watcher

Three genuinely different patterns depending on the office, because the offices themselves differ in what's available:
- **CIPO:** port the existing Python watcher already running nightly on the firm's AWS VPS (built by James in Claude Code) into the new worker framework, and extend its outputs into docketing, auto-filing, and draft-report generation.
- **USPTO:** notification-driven, not a scraper. Patent Center's nightly notification emails get recognized by M6 (email management), which triggers an ODP (Open Data Portal) API fetch for the structured status data and documents. TSDR is used for US trademark status as needed.
- **EPO / WIPO:** straightforward API polling against EPO OPS and WIPO's published interfaces.

## A2 — Task-Rule Builder

Converts natural-language rule descriptions into formal rule definitions plus test cases, gated behind approval before going live — the mechanism by which M1-R4's "rules are data, editable via natural language" actually works in practice.

## A3 — Rule-Change Monitor

Watches for practice-notice changes at the various offices and proposes rule diffs when something changes — so a CIPO or USPTO practice update doesn't sit unnoticed until someone happens to catch it manually.

## A4 — G&S Harmonizer

Takes a goods-and-services description coming in through a Madrid Protocol designation of Canada and maps it toward CIPO's pre-approved-terms Manual — grounded on the Manual dataset itself plus observed patterns of previously-accepted descriptions in the Canadian trademark database, so the suggestions are drawn from what's actually been accepted before, not just theoretically compliant.

## A5 — Search Strategist

Takes a technology description and produces IPC/CPC classification suggestions plus USPTO-syntax Boolean search strings — a starting point for prior-art searching, not a replacement for the searcher's judgment.

## A6 — Prior-Art Analyzer

A genuinely multi-step pipeline: given a folder of PDF references, it builds a bibliographic spreadsheet, highlights relevant passages against the technology description, drafts a novelty/obviousness analysis with claim-limitation suggestions, and produces a reporting letter from a template. Feeds directly into the M11 Prior Art & IDS module.

## A7 — Expense Matcher

Reconciles the bank feed against recorded expenses using heuristic matching, with a manual UI fallback for anything the heuristics can't confidently resolve. This is the engine behind M3's reconciliation loop.

## A8 — Email Instruction Detector (D15)

Reads ingested email across every mailbox in scope — not just the shared ones — and looks for something more specific than ordinary matter classification: **instructions to actually perform work**. A new engagement request, "please file the national phase entry," "please respond to the office action," "please renew this" — that class of message. When it finds one, it proposes either starting the intake chain (for a new client or matter) or creating a work item (for an existing matter), pre-filled with whatever parameters it could extract — client, family, jurisdiction, the requested action, any stated deadline. **Every single proposal requires human approval before any outbound step happens** — this is one of the clearest examples in the whole spec of the human-in-the-loop principle being load-bearing rather than decorative. Every detection is logged alongside the source message it came from, for auditability. As a false-negative safety net, anything A8 doesn't confidently classify still lands in the ordinary shared-mailbox triage queue rather than silently disappearing. Sender trust tiers (a known, established client contact versus an unknown sender) control how much of the proposal gets pre-filled versus flagged for closer review.

## A9 — Per-User Email Drafting Agent (D21)

One instance per team member, monitoring that person's own mailbox and drafting proposed replies **in their own writing style**, learned from their own historical sent mail, and grounded on the shared Knowledge Base (§ below) with citations back to the specific authority relied on — a MOPOP section, an MPEP section, whatever's actually applicable. Drafts land in that user's review queue (or as Outlook drafts) and are never auto-sent. The template for this agent is the owner's own existing personal email-drafting agent — the rollout plan is to generalize it, then duplicate and configure it per user. Code access via James remains a pending gate. A9 shares its email-monitoring infrastructure with A8 and A10 — one ingestion pipeline feeding multiple independent detectors and drafters, rather than three separate systems each re-reading the same mailboxes.

## A10 — Quote Agent (D22)

Detects quote requests in email by keyword signal (words like "quotation," "estimate," "cost" appearing alongside a PCT application number or a WO publication number) and drives the firm's existing website **Instant Quote tool** — already built, doing a Python-based retrieval of filing and priority data from WIPO, generating a fee-table-based custom quote, and emailing it out with both a link and a PDF attached. The OS integration layer adds: a quote record created in the OS's CRM with pipeline placement and matter/prospect linkage, a rollout to every inbox rather than just the one it currently monitors, and an extension to trademark quoting (the logic is simpler and the website tool already has it). Existing pipeline code access via James is a pending gate.

## A11 — OA Reporting Drafter (D23)

Typically triggered by A1 when an office action gets flagged. Reads the actual OA document and drafts the substantive section of the client-facing reporting email — a concise restatement of the objections raised plus proposed responses — written in the firm's own established voice, learned from the corpus of historical OA-reporting `.msg` emails (the same legacy corpus being migrated for M6 doubles as this agent's training and evaluation set). The draft gets inserted into a merge-field template that auto-populates the mechanical details (application number, filing date, applicant, title) straight from the database. Always a draft for professional review, never auto-sent — and its style and substance get evaluated against held-out historical reports before it's allowed to go live for real use.

## A12 — CIPO Filing Assistant (D24)

Invoked via the orchestrator chat (e.g., "file the 3DB-0007-CA application") or from a workflow task. Populates CIPO's online filing system from OS data — patent filing forms and document uploads as the primary focus, but also trademark filing (including the goods-and-services statement, which A4's harmonizer output feeds directly into) and industrial design filing (design-drawing PDFs plus description upload). Once the form is fully populated and everything's staged, **the agent stops** and creates a review task for the assigned patents or trademarks admin — who verifies everything and clicks submit themselves. This is not a configurable behavior; it's a hard rule. After submission, acknowledgment documents are captured automatically via M6 and auto-filed.

## A13 — USPTO Filing Assistant (D24)

The same populate-and-stage-then-stop pattern as A12, applied to USPTO's Patent Center. Forms that AppColl currently generates as fillable PDFs are instead regenerated by the OS's own document-generation layer, or Patent Center's native web-based forms are used directly where they're equivalent. Same review-task-then-human-submits flow as A12. Both A12 and A13 log every single field they write, for a complete audit trail, and neither ever stores portal credentials anywhere except Bitwarden, fetched fresh at runtime.

## A14 — Opposition Watch & Outreach (D25)

Runs against every weekly CIPO Trademarks Journal publication. Ingests the advertised marks (parsing the Journal itself, backed up structurally by the IP Horizons weekly TM XML data 🔎). **Mode (a), client watch:** runs similarity screening — string matching, phonetic matching, visual-element matching, plus Nice classification and goods-and-services overlap scoring — against the firm's watchlist, generating alerts, client reports from templates, and an automatically docketed opposition deadline (2 months from advertisement, extension-aware) via M1. **Mode (b), prospecting:** for every advertised mark, searches the Canadian trademark register for earlier marks owned by non-clients that could plausibly conflict, scores and ranks the conflicts, resolves who owns the earlier mark, enriches their contact details via the hunter.io API and LinkedIn manual-assist (consistent with D16's no-automation rule for LinkedIn), and drafts a personalized outreach message flagging the conflict and the closing opposition window. **Every outreach list and every message is reviewed and approved by a human before sending** — no exceptions. Sends are built to be CASL-compliant: identified sender, unsubscribe mechanism, and a recorded consent basis, generally preferring publicly-published business contact information where the outreach is genuinely relevant to the recipient's professional role. Everything is logged to the prospect ledger. Screening thresholds are tunable, and false positives feed back into improving the scoring over time.

## A15 — Self-Represented Applicant Prospector (D26)

Scans Canadian trademark application data (from the IP Horizons weekly TM XML diff, cross-checked against the Canadian Trademarks Database) on a biweekly or weekly cadence, matching the data's own refresh rate, looking for applications with **no trademark agent of record** that recently received an **Examiner's Report** — self-represented applicants who've just hit an examination objection they likely don't know how to handle. For each hit, a prospect record captures the mark, applicant, filing date, report date, and — critically — the response deadline recorded as the record's own expiry, plus the objection type where it can be derived. Contact resolution starts with the public application record itself before reaching for paid enrichment (hunter.io, LinkedIn manual-assist), and outreach runs through the exact same CASL-compliant, human-approved pipeline as A14. A14 and A15 deliberately **share one underlying TM Prospecting Engine** — the register ingestion, the prospect ledger, the enrichment adapters, the compliance gates, the outreach tracking — meaning any future prospecting campaign type is just a new detection rule bolted onto the same engine, not a new system to build.

## A16 — PCT National-Phase Prospector (D27)

Queries WIPO's PatentScope 🔎 (access-tier to be confirmed at build time) for PCT applications whose priority date puts the Canadian 30-month national-entry deadline inside a configurable window (defaulting to the next 3 months). For each candidate applicant, the pipeline runs: **(1)** a Canada-propensity check — does this applicant historically enter national phase in Canada at all, based on CIPO and PatentScope national-phase records; **(2)** an incumbent-firm analysis — which 3–5 Canadian firms has this applicant actually used, based on agent-of-record frequency on their past Canadian entries; **(3)** a ranking — Canadian filing volume over a configurable window (default 3 years) and the size of the firms they tend to use (large-firm users are natural price-arbitrage targets, small-firm users are natural service-quality targets); **(4)** the **reciprocity guard** — cross-checking incumbent firms against M7's reciprocity data and flagging or excluding applicants whose usual Canadian firm is one of Brunet & Co.'s own referral partners, on a per-firm, owner-controlled allow/block list, so this agent never accidentally cannibalizes a relationship that sends the firm reciprocal work; and **(5)** direct applicant contact resolution and human-approved outreach, with the approaching deadline itself as the hook, and the record expiring when the window closes. Designed to extend to a US-entry variant, and runs on the same shared prospecting engine as A14/A15, generalized beyond trademarks.

## A17 — IP Audit Agent (D28)

The analytical companion to A16, and also a standalone lead magnet in its own right. Triggered either by a website form (capturing the applicant's name and the requester's contact details into the CRM, queuing an audit) or a manual trigger from within the OS for a named applicant. **Canadian audit logic:** enumerate the applicant's full portfolio across PatentScope, EPO OPS, USPTO ODP, and CIPO; for each family, detect either a **granted US or EP equivalent**, or a **positive WO-ISA/IPRP opinion** on novelty and inventive step; identify which Canadian applications are still inside the **4-year examination-request window** (the population where this actually matters) and which pending or upcoming PCTs could still enter Canada; for every qualifying case, recommend the **Patent Prosecution Highway** and generate a summary of the granted or positively-opined claims — in effect, "here is specifically what you could obtain in Canada, faster and cheaper, because you've already done the hard work elsewhere." The whole audit rolls up into a report quantifying estimated cost savings (drawn from the fee-schedule service) and time-to-grant acceleration. A **US-facing variant** runs the identical family/work-product logic pointed at USPTO (Global PPH / PCT-PPH). Every audit is human-reviewed before it goes out, carries a general-information disclaimer (this is not legal advice), and both the requester and the applicant get recorded in the CRM with the audit itself as the first artifact in that prospect's pipeline. Claim summaries are LLM-generated directly from the actual claim text, with citations back to the source document — not paraphrased from memory.

## A18 — Reminder & Follow-up Agent (D31)

The firm's chasing engine, and the agent this project spent the most iteration on getting right. It's deliberately a *separate* agent that draws on the others — M1's deadline data, M6/A8's reply detection, A9's drafting capability, Xero's invoice data — specifically so staff never have to manually track which reminder is due to which client.

**(a) Deadline ladders.** Per task type and jurisdiction, an escalating sequence of templated reminders. The canonical maintenance-fee example: **T−2 months** (a courtesy notice), **T−1 month** (action explicitly requested), **T−2 weeks, labeled "FINAL REMINDER"** (stating plainly that absent instructions, the fee will not be paid and the application will be deemed abandoned by CIPO — with the equivalent USPTO expiry/surcharge language used for US matters). Every offset, every count, and every template's wording is fully configurable per task type and per jurisdiction, not hard-coded to this one example. The moment an instruction is received — whether A8 detects it in an email or someone logs it manually — the ladder halts and feeds directly into the annuity/task workflow. **A hard safety rule governs what happens if nobody responds at all: ladder exhaustion on a rights-preserving deadline never resolves silently.** It escalates to the responsible professional for an explicit, documented pay-or-abandon decision. The system is designed to never let a maintenance fee simply lapse by default.

**(b) Awaiting-client follow-ups.** Any work item tagged "awaiting client" — or prompted directly via the orchestrator — gets its own schedule (for example, every 7 days, capped at 3 reminders), with each reminder referencing the specific outstanding deliverable. A client reply halts the ladder and routes back to the item's owner; if the client's reply asks a question rather than just providing the answer, **A9 drafts a response for human review** before it goes out. If the ladder exhausts without a reply, the item escalates rather than simply closing — the same "don't let it go silent" principle as the deadline ladders, applied to ordinary client follow-ups.

**(c) Dunning.** Overdue-invoice statement ladders sent from accounting@, built from Xero balance data, escalating in tone, with the final stage carrying the stop-work warning described in M2-R8.

**Sending policy — the one part of this agent's design that changed mid-project.** The original design included a carve-out: fully deterministic, pre-approved templates could send automatically to verified contacts, on the reasoning that the content was known in advance and human review added nothing. **The owner later reversed this decision.** The final, current policy: **all reminders pass the human audit queue in v1**, reviewed and sent exactly like every other piece of outbound correspondence. The deterministic auto-send logic was not deleted — it's preserved, dormant, behind an `auto_remind` flag that can be set per client and per task type, defaulting to off. This means full automation, when the firm is ready for it, is a configuration change rather than a rebuild. AI-composed content (like A9's answers to client questions) always requires review regardless of how that flag is set. Every send, whichever path it took, is logged to the matter's correspondence timeline with delivery status, and unsubscribe/suppression preferences are always respected.

**(d) Stop-work integration.** When accounting's configurable thresholds trip — by outstanding amount, by age of the receivable, or both — the matter gets flagged: a color/badge change propagates across task lists, boards, capacity views, and My Day, so the flag is genuinely visible wherever anyone might be about to do work on that matter. New time or expense entries against a flagged matter raise a soft-block warning, which can be overridden but only with a logged reason. From that point, reminder and dunning language sent to the client includes the advance-payment condition explicitly. **Rights-preserving tasks — deadline responses, maintenance fees — are exempt from stop-work unless the principal explicitly signs off.** The reasoning is the same as the silence-never-abandons rule in reverse: a billing dispute must never be allowed to silently cost a client their legal rights.

## The Firm Knowledge Base Service (§12.1 — shared infrastructure, not an agent itself)

A single retrieval service — ingestion, chunking, embedding, citation-aware retrieval, and freshness tracking — consumed by Agents A3, A4, A9, A11, and the A0 orchestrator. Its corpus: the firm's own website and FAQ; for **Canada** — MOPOP, the Patent Act and Patent Rules, the Trademarks Act and Rules, the Trademarks Examination Manual, the Goods & Services Manual, and CIPO's website and practice notices; for the **US** — the MPEP, TMEP, 35 U.S.C., relevant Federal Register notices, the USPTO ID Manual (pre-approved goods/services), and the USPTO website; plus a deliberately **curated allow-list** of authoritative law-firm blogs and sites — curation here is intentional, both for quality control and for copyright hygiene, since retrieved third-party content is used for grounding and cited, never reproduced wholesale. Every source in the corpus carries version and refresh metadata, so an agent can flag if it happened to rely on an edition that's since been superseded.

## Prospect Engagement Orchestration (§12.2, D29)

This is the composition layer that lets several agents work together on a single prospect, semi-autonomously, from the very first touch through to the first filing — it isn't a new agent so much as a defined choreography between existing ones, owned by A0.

**Journey stages:** identified (by A14, A15, or A16 detection, or an A17 audit request) → contacted (an approved outreach message went out) → engaged (the prospect actually replied) → quoted (A10 generated a quote) → meeting (scheduling in progress) → converted (the intake chain from M7 kicked off) → matter opened → filing staged (A12/A13 populated and staged everything) → filed (a human clicked submit).

**How it actually works, mechanically:** every prospect has a conversation thread owned by the orchestrator; inbound prospect emails route to it automatically via M6's classification, matching against the prospect ledger by sender. If the prospect asks a question, A9 drafts a reply — grounded on the Knowledge Base, with citations, written in the assigned professional's style — and it goes into the audit queue. If the conversation surfaces a quote request, A10's detection logic fires in-thread, generates the quote from fee-table and register data, creates the CRM quote record, and attaches it to the drafted reply. When the conversation warrants scheduling a meeting, the draft can include a scheduling option — **Calendly**, which the firm already uses and has in its email signatures, and/or Microsoft Bookings or direct Graph free-busy lookups against the assigned professional's Outlook calendar — and a booked meeting creates both a CRM activity and a calendar event, linked back to the prospect record. An accepted quote, or simply a "go ahead" from the prospect (which A8's instruction detection applies to prospects exactly as it does to existing clients), triggers the standard intake chain: conflicts check, DocuSign ToE, Xero retainer to trust, family and matter opened — and at that point the journey formally hands off from marketing to production, ending in an A12/A13 staged filing that's waiting on a human to click submit.

**The human audit queue is non-negotiable in v1:** every outbound message, every quote, and every scheduling proposal is approved by a human before it sends. The design supports configuring autonomy per message type later (for instance, eventually relaxing review on purely FAQ-style replies to spot-checking rather than full review), but the default — and the entire v1 build — is full review of everything. Conversion and filing steps keep their existing, separately-specified human gates (D19 for signatures, D24 for filing) regardless of any future relaxation of the messaging gates.

**Safety rails, specifically:** rate limits per prospect, so no AI-driven pestering can happen even accidentally; automatic suppression the moment someone unsubscribes or opts out, per CASL; automatic escalation to a human the moment a thread starts looking like it's seeking actual legal advice rather than staying within marketing scope; sentiment and complaint detection that pauses a journey rather than pushing through it; and full thread logging throughout.

**The Agent Activity Dashboard**, which lives in the CRM: a per-prospect journey view (current stage, the thread itself, the next action, any expiry); a firm-wide funnel across all stages with conversion rates broken out per campaign type (A14, A15, A16, A17 each tracked separately); audit-queue workload and approval latency, so bottlenecks in the human-review step are visible; per-agent performance metrics (drafts produced, and their approval/edit/reject rates, response rates, quotes generated versus accepted, meetings booked, revenue ultimately attributed); and alerts for journeys that have stalled or windows that are about to expire. The funnel and revenue-attribution numbers feed directly into the EOS scorecard described in the M9 file — marketing performance sits inside the same weekly accountability rhythm as everything else, not off in its own silo.
