### 2. Auth & RLS — **[ruling, D44]** user-JWT pass-through to Postgres

RLS is the real access control, not defense-in-depth. The API must **never** use the Supabase service-role key on user-facing request paths.

- SPA authenticates against **Entra ID** (MSAL, auth-code + PKCE). Create the Entra app registration (SPA redirect + API scope); document tenant/app IDs in `infra/entra.md` as placeholders.
- FastAPI validates the Entra access token, then **exchanges it per-request for a Supabase-compatible JWT** signed with the Supabase JWT secret: `sub` = OS user UUID, `role` = `authenticated`, plus a claims block reserved for WP 0.8 (family ACLs, D43 permission domains, D39 mailbox identity). Put the bridge in `py-shared/auth.py` with a request-scoped Supabase client factory.
- Service-role key is permitted only for: migrations, system workers acting with an explicit system identity, and admin scripts — each usage logged and enumerated in `DECISIONS.md`.
- **Proof-of-concept required in this WP:** one demo table with an RLS policy, two test users, and a pytest that connects **directly to Postgres** with each user's JWT and proves row isolation (this is the pattern the WP 9.1 permission acceptance tests will reuse). Wire it into CI.

### 3. Typed contracts

- FastAPI is the schema source of truth. `make gen-contracts` exports OpenAPI and regenerates `packages/contracts` (recommend `openapi-typescript` + a thin typed fetch wrapper, or Orval — team's choice).
- CI fails if generated contracts are stale (regenerate and diff).
- Ship one demo endpoint (`GET /api/v1/health` returning a typed payload) consumed end-to-end by the SPA to prove the loop.

### 4. MCP server surface — **[ruling, design review §4/§7.5]**

Expose the OS API as an **MCP server** from day one, generated/derived from the same OpenAPI schema so it never drifts. Skeleton only: server boots, serves the health/demo tool, auth deferred to a documented TODO tied to the D44 JWT bridge. Lives in `apps/api` as a mounted sub-app or sibling process — your call, record it.

### 5. Platform services, not 19 agents — **[ruling, D38/design review §5]**

The agent roster (A0–A18) is a spec taxonomy. Buildable architecture is ~6 shared services plus orchestrator and KB (laid out in §1 above): email ingestion/classification (A8/A9/A10/M6), drafting (A9/A11/A18/M8), prospecting engine (A14–A17), ladder/scheduler (A18), watcher framework (A1/A3), browser automation (A12/A13). In this WP each is a package with a README stating its purpose, consumers, and interface sketch — **no implementations**. Individual "agents" will later be configuration + prompts on these services.

### 6. Workers & events

Scaffold one worker process with a Postgres-backed job/event mechanism (recommend a plain `events` table + `FOR UPDATE SKIP LOCKED` poller, or pgmq — team's choice; no external broker). Demo: an event enqueued by the API is consumed by the worker and logged. Event types will grow at WPs 1.x/3.3/4.x.

### 7. Secrets — Bitwarden pattern (D10)

No secrets in the repo or in committed env files. `py-shared/secrets.py` loads at runtime from **Bitwarden Secrets Manager** (`bws`), with a documented local-dev fallback (`.env.local`, gitignored). CI uses repository secrets. Seed the Bitwarden project structure doc in `infra/secrets.md`.

### 8. Supabase & migrations

- `supabase` CLI for local dev (`make dev` boots local stack + API + web).
- Migration #1 creates the demo RLS table from §2. Convention: **every table ships with its RLS policy in the same migration — no "temporarily open" tables (D43).**
- Reserve (comments/doc only, no schema yet — WP 0.8 owns it): permission-domain grant tables, role templates (Agent/Paralegal/Bookkeeper/Principal), and the minimal **work-item substrate** (needed by WP 2.5.1 earlier than its phase slot — see tracker risk note).

### 9. Frontend

Vite + React + TS, strict mode. MSAL auth wired to the demo endpoint. **Browser-native constraint (§2 of spec):** every workflow must work in a plain browser — no Electron, no desktop dependencies, keep bundle discipline in mind from the start. Component/library choices are the team's (record in `DECISIONS.md`).

### 10. CI

GitHub Actions (assumed; substitute freely): lint (ruff + eslint), typecheck (mypy or pyright + tsc), tests (pytest + vitest), contract-drift check (§3), migration dry-run against a fresh Postgres, and the D44 RLS proof test. All green on the scaffold PR.
