## 1. Vision & Objectives

A unified OS that (1) runs the daily practice — docketing, deadlines, tasks, time & billing, expenses, documents, email, CRM — replacing AppColl as system of record; (2) embeds EOS discipline (scorecards, Rocks, issues, accountability, L10 cadence); (3) automates status monitoring, task generation, expense reconciliation, invoicing, reporting letters, intake, annuity instructions, and content publishing; (4) is agent-native and extensible; (5) integrates with M365 and Xero.

**Success criteria:** zero missed deadlines; 100% of official-fee spend matter-linked and invoiced at actual cost; email-to-work-item automation with human approval; single pane per matter; auto-populated weekly EOS scorecard; validated AppColl cutover.

---

## 2. Principles & Constraints

- **Integrate-first:** Teams (chat), Outlook (transport), SharePoint (storage), Xero (ledger), MailChimp (sending), DocuSign (signatures), Clarivate (annuity payment), WordPress (site). Build the connective tissue, not the utilities.
- **Family is the spine:** Technology Family → per-jurisdiction Matters, matching file references, SharePoint tree, and the future chain record.
- **Human-in-the-loop:** agents draft and propose; humans approve anything sent, filed, invoiced, instructed, or posted (v1). This explicitly includes A8 instruction detection (proposes intake/work items; never auto-sends ToE or retainer) and annuity instructions.
- **Auditability:** versioned rules; immutable logs of docket changes and agent actions.
- **Confidentiality:** sandboxed data; encrypted LLM egress (D8); Bitwarden secrets (D10).
- **Multi-currency:** CAD base; USD/EUR/CHF routine; actual-settled-rate re-billing (D12).
- **Bilingual awareness:** CIPO correspondence/data may be French.

---

## 3. Architecture

Monorepo. **FastAPI (Python)** API + **event-driven workers** (email arrival → classify/detect; status change → docket; payment webhook → apply trust; schedule → poll/refresh) + **agent layer** (Claude API) + **React/TS SPA** + **Supabase Postgres** with row-level security enforcing family permissions. Entra ID SSO. Typed API contracts generated from OpenAPI. Integration adapters per external system; credentials fetched at runtime from Bitwarden.

### 3.1 Data-Access Abstraction & Blockchain Fork (unchanged from v0.2)
Repository pattern (`FamilyRecordStore`); versioned canonical **Family Record** JSON schema (family ID, title, applicant/owner, per-country records: country, numbers, dates, status, responsible attorney/associate); family ACLs (applicant ultimate authority → initiating attorney delegate → per-country associate read grants) enforced in-app now, cryptographically later; signed `FamilyRecordExport` API usable immediately for associate sharing and later as the chain-write path. Chain itself = separate future project (Tracker F.1). Operational data (time, billing, email, EOS) stays off-chain permanently.

- M1-R3: AppColl rule library import as seed.

- M1-R5: Docket views + daily docket email.

- M1-R9: **Smart Matter Opening (D32):** matter-opening form asks jurisdiction + number → retrieves and populates bibliographic data from the authoritative office source (takeovers and PCT/Madrid/Hague national entries); child filings select parent(s) → relationship recorded and shared fields inherited; retrieved/inherited data presented as a review-and-confirm diff before save. Guided Q&A flow per D33 (the form asks only what retrieval can't supply).
- M1-R10: **Task-level fee/expense caps** (per-client cost-control ceilings surfaced in M2 at time/expense entry and billing).
- M1-R11 (D35): Tasks carry **dual dates (RespondBy + FinalDueDate)** plus trigger RefDate and ClosedOn; every task has a **DeadlineType ∈ {Hard External, Extendable External, Internal, General Reminder, Event, Transient Event}** (D37) driving A18 ladder behaviour, escalation severity, and the silence-never-abandons rule (Hard/Extendable only); task statuses include Received, Not Needed, and Missed (deadline passed, extension required) alongside Open/Completed.

- M1-R13 (D37): **Rule-engine capabilities from the legacy library (552 imported rules):** owner resolution supports fixed role queues, matter-relative roles (attorney/paralegal for matter), current-user, and multi-assignee expansion (all inventors); rules may define **alternate offset paths** (conditional dual deadlines); rules may include **actions** beyond task creation — matter-field setters with template expressions over the triggering task/event; auto-generation toggles per rule; legacy reminder-pair task types migrate into A18 ladder definitions rather than surviving as standalone tasks.

### 7.1 Annuity Instruction Workflow (M1-R8 + Clarivate)
Deadline approaches → client report/instruction step (pay/abandon, response recorded) → instruction to Clarivate → confirmation reconciled → invoice to M3.
🔎 Clarivate's documented renewals integrations are with its own IPMS products (FoundationIP/IPfolio); no public third-party instruction API found, but bespoke firm integrations exist (e.g., J A Kemp) — a negotiated file/feed channel is plausible.

## 12. M9/M10 — Workflow, EOS & Agents

### 12.0 Work Management (D30)

**Objective:** all work — scripted, unscripted, and intra-day — lives in the OS; email and Word cease to be workflow tools. Design emulates Monday.com's interaction model and dedicated EOS software's meeting mechanics, natively and matter-linked.

**Three-tier work model (one engine, one queue):**
- **Tier 1 — Docket tasks:** rule-generated, deadline-driven (M1). Immovable dates; escalation on risk.
- **Tier 2 — Projects:**
  - *Scripted:* template-driven with predefined stages, chained tasks, role routing, and standard cycle times — patent application drafting, OA responses, national entries, TM prosecution, oppositions. Templates versioned and editable.
  - *Unscripted:* ad-hoc projects (e.g., assignment recordal across N jurisdictions, obtaining signatures for a deceased inventor's estate). Created manually or via the **orchestrator: describe the project in natural language → draft plan (stages, tasks, owners, dependencies, target dates) → user edits → launch**. Creating a tracked project must be easier than opening Word.
- **Tier 3 — Micro-requests:** intra-day review/approve/answer items created by **@request on any work item or document** ("review this response before filing"). Each carries an SLA timer, notifies via Teams, **blocks its parent until resolved**, supports unlimited same-day round-trips, and logs turnaround. This is the designed replacement for email ping-pong.

**Board framework (Monday-model, native):** configurable boards over work items with typed columns (status, owner, due, priority, matter, stage, custom), **views: table · kanban · timeline/Gantt · calendar · workload**, groups, saved filters, subitems; every item matter/family-linked at creation. **User-configurable automations** ("when status → Filed, create invoice-review task for accounting; when review approved, unblock parent and notify owner") extend the workflow-template engine into no-code territory.

**My Day:** each user's single unified queue — docket tasks + project tasks + micro-requests + EOS To-Dos — priority-ordered (deadline proximity, SLA, priority, blocked-status), with quick actions (complete → chained task spawns; request review; defer with reason). Daily digest replaces the morning inbox triage.

**Assignment & capacity:** suggestions/auto-assignment from role, live workload, and historical cycle time per user per task type; interactive capacity board (per-person, per-matter, drag-to-reassign) replaces the Word Patent/Trademark Project Lists; SLAs and escalation to responsible professional + firm dashboard.

**Productivity visibility (seat-owned measurables, not surveillance):** per person — on-time completion %, throughput by task type, cycle time vs. task-type baseline, micro-request turnaround, aging WIP, To-Do completion rate (EOS 90% target tracked explicitly); per firm — funnel of open/at-risk/overdue, docket compliance, review-bottleneck detection. Metrics surface on the person's own dashboard first, then aggregate.

**Weekly accountability reporting:** auto-generated **L10 pack** (firm scorecard with red/yellow/green, Rocks status, To-Do completion, overdue/at-risk items by owner, aging WIP, next-7-day critical deadlines, headline metrics) drives the L10 meeting mode; **per-person 1-on-1 reports** (same metrics scoped to the seat + their open items) support supervision conversations. Both exportable/printable.

### 12.0.1 EOS Layer

### 12.2 Prospect Engagement Orchestration (D29)
The composition layer that lets the agents work **together** on a prospect, semi-autonomously, from first touch to first filing:

**Journey stages:** identified (A14/A15/A16 detection or A17 audit request) → contacted (approved outreach) → engaged (prospect replied) → quoted (A10) → meeting (scheduling) → converted (intake chain, D19) → matter open → filing staged (A12/A13) → filed (human submit).

**Mechanics:**
- Every prospect has a **conversation thread** owned by the engagement orchestrator (an A0 capability, not a new agent): inbound prospect emails route to it via M6 classification (prospect-ledger sender matching).
- **Query responses:** A9 drafts a reply — grounded on the Knowledge Base with citations, in the assigned professional's style — into the audit queue.
- **Quote requests:** detected in-thread (A10 logic), quote generated from fee tables + WIPO/register data, quote record created in CRM, quote attached to the drafted reply.
- **Meeting proposals:** when the conversation warrants it, the draft includes scheduling — **Calendly (incumbent, already in the firm's signatures) and/or Microsoft Bookings / Graph free-busy** against the assigned professional's Outlook calendar; booked meetings create CRM activities and calendar events with the prospect record linked.
- **Conversion:** an accepted quote or "go ahead" (A8 instruction detection applies to prospects too) triggers the intake chain — conflicts → DocuSign ToE → Xero retainer to trust → family/matter opened — and the journey hands off from marketing to production, ending in an A12/A13 staged filing awaiting the human submit.
- **Human audit queue (non-negotiable v1):** every outbound message, quote, and scheduling proposal is approved by a human before sending. Autonomy is **per-message-type configurable** (e.g., later relaxing FAQ-style replies to spot-check) but defaults to full review; conversion and filing steps always retain their existing human gates (D19, D24).
- **Safety rails:** rate limits per prospect (no AI-driven pestering), suppression on unsubscribe/opt-out (CASL), automatic escalation to a human when the thread shows legal-advice-seeking beyond marketing scope, sentiment/complaint detection pausing the journey, and full thread logging.

**Agent Activity Dashboard (CRM):** per-prospect journey view (stage, thread, next action, expiry); firm-wide funnel (identified → contacted → engaged → quoted → meeting → converted → filed) with conversion rates per campaign type (A14/A15/A16/A17); audit-queue workload and approval latency; per-agent metrics (drafts produced, approval/edit/reject rates, response rates, quotes generated/accepted, meetings booked, revenue attributed); alerts for stalled journeys and expiring windows. Funnel and revenue-attribution measurables feed the EOS scorecard (M9).

## Appendix A — Reference & Directory Conventions (complete)

**Client code:** short alpha (e.g., `3DB`).
**Family:** `{ClientCode}-{FamilySeq}` (4-digit, per client), folder `{FamilySeq} - {Technology Title}`. Trademark families tagged after the number: **`(TM)`** standard-character mark, **`Design`** design mark; patents untagged. Matter types not otherwise encoded in the reference.
**Jurisdiction segment (ordered sequence per family/country):**
- `USP` — US provisional (priority filing) → `US` — first regular filing → `US2`, `US3`… — continuations/CIPs/divisionals (type held in `relationship_type`, not the reference). **Observed in production: sequences to `US11`, `CA5`, `EP2`, `AU2`, `MX3`; the sequel pattern also applies to trademark extension applications.**
- Two-letter codes for national filings (`CA`, `US`, `GB` [incl. auto-created Brexit-clone TMs], `AU`, `JP`, `CN`, `IN`, `KR`, `MX`, `ZA`, `TW`, `NL`, …) — the docket covers many foreign-associate jurisdictions, not just the fee-table offices.
- `PCT` — WIPO/PCT patent vehicle; `MP` — Madrid Protocol trademark vehicle. **Both exist as separate sibling matters; national designations open at the same level (not nested).** Data model links designations to their vehicle via `parent_matter`.
- **No-jurisdiction matters (D35):** general/advisory work (IP strategy, NDA review, searches) uses `{Client}-{Seq}` with no country segment; client-level catch-alls use `{Client}-9999`-style refs. OS models these as **Advisory/General matter types** at family level.

```
    Account/                               per-client accounting docs (sibling of families)
```
**Design rulings from the crawl (D36):** (1) folder-name **status/stage annotations duplicate docket state in the filesystem** — in the OS, status lives in the database; the M5 mapper parses/tolerates `(...)`-suffixed legacy folder names when binding folders to matters, and **OS-created folders omit status annotations** (no rename churn; status visible in-app) — owner confirmed; (2) foreign-associate firms appear as clients (dual-role organizations) — the CRM models one organization with both roles, and these codes feed the D27 reciprocity guard; (3) legacy family/jurisdiction folders map to matters via tolerant pattern-matching with a manual-resolution queue for non-conforming names.
